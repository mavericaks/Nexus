# 0001. Modular monolith, not microservices

Status: Accepted

Context: Full microservices decomposition is not the current industry default it's often assumed to be — Amazon Prime Video and Segment both publicly reversed distributed designs back into monoliths after finding the distributed version solved problems they didn't have. A solo, free-tier project decomposed into many services would also outstrip the Neon/Upstash/Render free-tier budget and the 8GB K8s cluster budget already committed elsewhere in this project.

Decision: Nexus is built as a single modular monolith with Clean Architecture boundaries (domain/application/infrastructure/api), with exactly two deliberate exceptions: the MCP tool server (Phase 4) and the Notification Service (Phase 5). Both are extracted because they have a genuine architectural reason — the MCP server exists to prove the protocol boundary is real, and the Notification Service has no synchronous dependency on the core API, consuming only Kafka events.

Consequences: Extraction elsewhere remains cheap if ever justified, since the domain/application boundaries already don't leak infrastructure concerns. No other service gets pulled out without a documented reason meeting the same bar as these two — "microservices because senior projects have them" is explicitly not a valid reason on its own.
